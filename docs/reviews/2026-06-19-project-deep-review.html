CAL-1 New calibration wizard never persists to disk — confirmed calibration lost on restart CriticalValidatedcalibration
orin/wavecam/wavecam/control_calibration.py — _commit_location:445, check_level:477, validate_heading:604, confirm_validation:632

The entire session-based wizard (/calibration/session/*, /location, /level, /heading-lock, /validation, /validation/confirm) mutates self.pipeline.pose (which is the store's pose object in memory) and self._session, but none of these methods call self._store.set_step() or self._store.save(). A repo-wide grep for save()/set_step in this file returns only the FOV path (:723) and the legacy capture_calibration (:808). The final confirm_validation only flips in-memory flags (valid=True, banner="VALID").

Consequence: an operator runs the full wizard, sees "VALID", restarts the box — and CalibrationStore.load() reads the old file: reference_heading=None, banner INVALID. This silently re-introduces the exact "gps_calibrated true but reference_heading null" class CalibrationStore was created to prevent. (The live rig currently shows reference_heading=40.5 only because that was set via the legacy route, which does persist.)

Each wizard commit step should call self._store.set_step(<step>, entry) then self._store.save() under the lock — exactly as capture_calibration does. Map heading→reference_heading (the store already does this). Return a 503 on save() failure (mirror the FOV path at :723) so the operator never sees ok:true for an unsaved calibration.
IOS-1 ShadowDetail fields silently decode to nil — snake_case CodingKeys vs global convertFromSnakeCase CriticalValidatediOS
ios/WaveCam/Sources/WaveCamClient.swift:748-755 (decoder at :1843)

The shared decoder uses .convertFromSnakeCase, which converts bearing_deg → bearingDeg and then looks up a CodingKey whose rawValue is "bearingDeg". ShadowDetail declares explicit CodingKeys with literal snake_case raw values (case bearingDeg = "bearing_deg") — so the converted key never matches and every shadow-event field decodes to nil. This is the precise gotcha the file's own comments (lines 118-123) warn about, re-introduced in this one struct. Result: the iOS shadow/estimator readout is silently blank.

Delete the CodingKeys enum from ShadowDetail. The camelCase property names (bearingDeg, distM, …) then auto-match the converted keys — no other changes needed.

API-1 agent_chat runs a ≤120s subprocess on the shared Starlette threadpool — can starve /safety/kill HighValidatedcontrol API / agent
orin/wavecam/wavecam/control_api.py:718 (def agent_chat — synchronous) · control_system.py:128-150 · agent_session.py REQUEST_TIMEOUT_SEC=120

agent_chat is a plain def, so FastAPI runs it on the shared threadpool (default 40 workers), each call blocking a worker for up to 120 s (the claude -p timeout). Enough concurrent/retried chat turns can exhaust the pool and stall every /api/v1 route — including /safety/kill. The arm-TTL clamp protects the subprocess duration, not threadpool occupancy. This is the highest-blast-radius finding: a degraded agent turn can indirectly delay a safety command.

Make agent_chat async def and run the blocking chat() via await asyncio.to_thread(...) behind a bounded Semaphore (e.g. 2), so safety/control routes keep their capacity. At minimum, document that the threadpool size must exceed expected concurrent agent turns.
SAFE-1 STOP command can be de-dupe-suppressed for up to 0.25s on the killed path HighValidatedPTZ / safety
orin/wavecam/wavecam/pipeline.py:234-253 (_send_cmd) · killed loop routes STOP at :656

Send condition is changed or (due and not cmd.is_stop) or stop_due. Once a STOP has sent and re-asserts within stop_resend_interval (0.25s) with changed=False, it is silently dropped. The killed-loop re-issues STOP every frame, so a PTZ that creeps/drifts after the first stop isn't re-stopped for up to 250 ms — uncommanded motion on a safety path.

On the kill-latched branch, bypass de-dupe and call self.ptz.stop() directly (as kill() already does), or force-send any cmd.is_stop while killed.
SAFE-2 claim_manual takeover still uses release-then-request (the race transition() was built to close) HighValidatedPTZ / safety
orin/wavecam/wavecam/control_ptz.py:51-62 (claim_manual takeover branch), and :159-164 (hold_manual_owner)

The takeover path does release(current_owner) → stop/zoom-stop → request("manual") as separate steps. The whole block holds the dispatcher lock, but the arbiter runs on the pipeline thread and reaches owner.transition() through the owner lock only — so in the window after release (owner=IDLE) and before request("manual"), the arbiter can transition(IDLE, vision_follow) and seize the PTZ. There is a restore-on-failure fallback (:60-62), but it restores a stale current_owner. This is the one place still doing release-then-request instead of the atomic transition().

Replace the release→request pair with a single self.pipeline.owner.transition(current_owner, "manual") (guarded by current_owner in AUTONOMOUS); on failure the owner is unchanged, so no restore is needed. Do the same in hold_manual_owner.
SAFE-3 send_manual_velocity does not re-check killed before issuing VISCA → one frame of motion after KILL HighValidatedPTZ / safety
orin/wavecam/wavecam/control_ptz.py:166 (send_manual_velocity)

The route checks killed, then calls claim_manual (which itself re-checks under the owner lock — good), but send_manual_velocity does not re-check killed before self.pipeline.ptz.pan_tilt(...). A KILL arriving after claim_manual succeeds but before the VISCA bytes go out still moves the camera for one frame.

Re-check self.pipeline.owner.killed at the top of send_manual_velocity/send_manual_zoom_velocity; short-circuit to ptz.stop() if killed.
GPS-1 DirectRadioGps erases the last good target fix on any no-fix packet HighValidatedGPS
orin/wavecam/wavecam/gps_direct_lora.py:194-220 (_handle_remote_line)

When a line arrives without the fix flag (surfer indoors, cold start, LoRa drop), fix stays None and line 219 runs self._latest = fix → self._latest = None. The next get_fix() returns None instantly, so the drive_stale_sec age-gate never gets to age out a stale-but-usable fix — the track drops on a single no-fix packet. This contradicts the carefully documented "keep last-known-good" comment at :197-205 (which only guards the corrupt-coords-with-fix case). Legacy/Meshtastic sources hold the last fix and let the age gate decide.

Only overwrite self._latest when fix is not None. Still update _target_telemetry and _last_poll_ts (battery/sats are valid without a fix); let get_fix() re-age the retained fix so the existing age gate drops it cleanly once truly stale.
GPS-2 lock_location live-base fallback has no freshness gate — can lock a stale (rebooted-Wio) position HighValidatedcalibration / GPS
orin/wavecam/wavecam/control_calibration.py:335-346 (live-base fallback) · gps_direct_lora.py:242-244

When the client sends no samples, the wizard falls back to gps.get_camera_position(), which returns the last cached (_cam) tuple unconditionally — no timestamp, no age. get_camera_age() and reader_alive() exist but aren't consulted. Per the known gotcha (base Wio reboot silently staleifies the serial), a rebooted Wio leaves a stale _cam cached; this path would lock it as the camera location with a 15 m radius, silently corrupting every subsequent bearing. The sample-based path above it has a max_fix_age_sec gate; the fallback bypasses all of them.

Require gps.reader_alive() and gps.get_camera_age() below a tight bound (2-3 s) before accepting the live base; else return location_unavailable/gps_stale. Pass fix_age_sec into the synthesized sample so the existing age gate applies.
VIS-1 Color-only blob can acquire full lock while GPS owns (gps_boost crosses lock_threshold) HighReviewer-reportedvision / fusion
orin/wavecam/wavecam/fusion.py:179-193 (_select) + pipeline.py:643-649 (_gps_cue)

With defaults (require_person=False), a color-only blob yields CONF_SUSTAIN≈0.45, below lock_threshold=0.60 — so it does not self-lock when vision owns. But while GPS owns, the legacy frame-center cue still fires and +gps_boost (0.2) is added → 0.65 ≥ 0.60 → an orange object near frame center (buoy, sun glint, trash) can self-lock and steal the camera to vision_follow. The fusion docstring's "a random orange object cannot self-lock" claim holds only when GPS is not directing.

Require a confirmed person (matched=True) for the GPS boost to cross lock_threshold, OR cap the boosted color-only path strictly below lock_threshold (e.g. min(CONF_SUSTAIN + boost, lock_threshold - ε)) so GPS can sustain an existing lock but never acquire on color alone.

Status note: the logic is traced through the code; not live-confirmed against a real buoy. Worth a bench test before the next water run.

ZOMBIE-1 Zombie-rig defense is observe-only — a dead vision loop keeps reporting TRACKING with fps=0 HighValidatedvision / safety
orin/wavecam/wavecam/tracking_arbiter.py:99-104 (decide) · health.py:20-29 · control_api.py /health

HealthRegistry turns heartbeat silence into ok:false in /health — but nothing acts on it. decide() only checks self.enabled (the DISABLE-PTZ latch); it does not check capture/detector liveness (confirmed: grep for health/beat/capture/detector in tracking_arbiter.py returns nothing relevant). So when the vision thread dies but the API stays up, /status keeps reporting state:TRACKING with fps:0 and the PTZ keeps being driven by stale fusion. This is exactly the two-rig incident class. /health alarms; the operator HUD and supervisor do not.

In tracking_arbiter.decide() (or the pipeline frame loop), gate vision ownership on health.snapshot()["components"]["capture"]["ok"] (and detector). When stale while mode != gps_only, force owner="idle" and emit a vision_stale event — same pattern as the existing base_drift revalidation gate.
KILL-1 kill() does not reset arbiter vision hysteresis — RESUME can instantly restore a stale false lock HighReviewer-reportedvision / safety
orin/wavecam/wavecam/pipeline.py:215-232 (kill) · tracking_arbiter.py (hysteresis state)

kill(on=True) latches PTZ + state.killed; kill(on=False) resumes and leaves owner idle "for the arbiter to re-decide." But the arbiter's _consecutive_locked/_last_locked_time/_vision_owns are not reset on kill. If kill happened mid-lock, the first post-resume frame with locked=True re-grants vision_follow instantly — even if the operator killed precisely because the lock was wrong. (Manual release correctly resets via reset_vision_state at control_ptz.py:98 — see rejected FP-1; KILL does not.)

In kill(on=True), call self.arbiter.reset_vision_state() so post-resume acquisition follows the full K-frame hysteresis.

VIS-2_send_cmd can drop a resume-after-STOP within the 50ms command intervalMedReviewer-reportedvision
pipeline.py:234-253

EST-1Estimator scalar covariance update is non-symmetric (can break PSD over time)MedReviewer-reportedestimator
orin/wavecam/wavecam/estimator.py:414-432 (_scalar_update)

The update does P[i][j] -= KhP[i][j] * P[j][j] — only the diagonal in the right factor. The docstring admits "approximate but stable," and there is a max(0.0, ...) guard at :462 (a tell that negative variance was contemplated). At 35 fps with GPS+vision+ bearing+range all hitting _scalar_update, asymmetry accumulates; first symptom is a negative var_brg_rad or NaN.

Replace with the full (I-KH)P form (4×4, cheap — helpers exist), or assert P[i][i] >= 0 after each update and reset on violation. (Math not independently re-derived — flagged for a numerics pass.)
EST-2Vision bearing innovation uses a linear small-angle approximation across the full FOVMedReviewer-reportedestimator
estimator.py:338-348 + the pixel_offset_deg = (cx-w/2)/w * fov term

The pixel→bearing offset is linear (cx-w/2)/w*fov, but the true angular offset is atan-shaped. At the current narrow FOV the error is ~1° (negligible), but with r_vis_deg=1.0 (tight) it can bias the filter, and it's a latent trap if the FOV widens or the aim point crops off-center.

Use the pinhole form: degrees(atan((cx-w/2)/(w/2) * tan(radians(fov/2)))).
IOS-2WCConfig is not tolerantly decoded — one missing field kills the whole Tune panelMedReviewer-reportediOS

DEPLOY-1deploy.sh has no rollback path + a fixed sleep 12 windowMedValidateddeploy
orin/wavecam/deploy.sh:28-37

What is solid (confirmed correct)

  • KILL supremacy — PtzOwner.kill() is sticky, fail-closed, re-checked under lock in every mutating call; nothing re-arms/re-owns after kill without resume(). The agent's separate KILL latch clears only via agent_resume().
  • Auth roles per route — all correct: /safety/*=SAFETY, /agent/chat|summon=SERVICE, /agent/arm=CONFIG, PTZ moves=PTZ, calibration captures=PTZ, calibration/fov POST=CONFIG/GET=READ, media read/download=READ, record/delete=CONFIG, restart=CONFIG. No route has a wrong role.
  • No silent-200 refusals — api.refusal() defaults to HTTP 409; every caller passes an explicit non-200 for non-conflict cases. ok:false always carries a non-2xx status.
  • Refusal checks — callers that branch on refusal (apply_hot_config, preset apply, calibration captures) all check correctly.
  • Hot-config persist — writes post-coercion values to config.local.yaml (atomic os.replace, under a lock), never the main YAML; round-trips; rsync excludes it.
  • _KNOWN_SECTIONS complete — all 13 sections present (incl. agent); the silent-vanish gotcha is closed and tested.
  • Fusion hysteresis guard — refuses unlock >= lock at load (after overlay) and in hot-config; the 2026-06-11 permanent-lock failure is defended.
  • Geometry — haversine/bearing/elevation/normalize_180 textbook-correct, units consistent (deg on the pose seam; e7/centi conversions correct at the radio boundary); the pan_encoder_to_bearing inverse is algebraically exact (the prior zombie-bug inverse is genuinely fixed).
  • Deadman timers — cancel-and-bump-then-start under lock; generation-token checked before acting; held manual lock survives deadman expiry.
  • OAuth token handling (agent) — never on argv (stdin), never logged, redacted from error messages; KILL re-reads can_act() after the turn; arm TTL clamped ≥ timeout+60.
  • iOS Codable discipline — all 23 init(from:) live in extensions (memberwise inits survive); every GET goes through getWithFallback; the 1 Hz poll never wipes a failed-KILL message (lastCommandError separate from lastError). (The one exception is IOS-1 / ShadowDetail.)
  • mypy gate scope — intentionally the 6 pointing-math files (the 2026-06-11 zombie-bug class); CI matches.
  • Tests — no test asserts on the live rig; every rtsp:///127.0.0.1 hit is a string to a stub; conftest.py isolates camera_pose.json.

Top test-coverage gaps

  1. Calibration wizard persistence (CAL-1) — no test restarts and reads the pose file back through the wizard path. test_calibration_persist.py only covers legacy capture_calibration.
  2. Zombie → KILL/downgrade gate (ZOMBIE-1) — no test that a stale capture heartbeat while state==TRACKING forces owner=idle.
  3. supervisor.restart_systemd_unit + allowlist guard — zero tests; the one restart path in the system and the safety fence (unit != "wavecam.service" → ValueError) is unasserted.
  4. START AUTO vs DISABLE-PTZ latch interaction — no test for /ptz/auto while tracking.enabled=False (code: start_autonomous succeeds, owner rolls back next frame → a flashing AUTO indicator with no tracking).
  5. YAML/overlay type coercion (CONFIG-1) — no test that quoted "false"/"10" load as the right type.
  6. deploy.sh — no shellcheck/bats/smoke; the rollback + version-fetch-failure paths are entirely manual.
