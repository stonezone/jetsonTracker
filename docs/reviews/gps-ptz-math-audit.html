High Target altitude is hard-coded to sea level in live GPS pointing

orin/wavecam/wavecam/gps_pointing.py:50–61 and orin/wavecam/wavecam/gps_stub.py:17–25

What is wrong: NormalizedFix carries no altitude field. compute_target builds the target GeoPoint from only lat/lon/speed/course, so target.alt_m defaults to 0.0. The latched base altitude comes from the base Wio (which may be on a bluff or cliff). elevation_deg therefore computes atan2(0 - base.alt_m, distance) instead of atan2(target.alt_m - base.alt_m, distance).

Why it matters for aim accuracy: Any time the base is above or below sea level while the tracker is in the water, the tilt command is biased by roughly atan(base_alt / distance). At 100 m range and a 10 m bluff the camera tilts down ~5.7° more than it should, easily putting the surfer out of frame at tele zoom.

Proposed fix

# 1. Add altitude to the normalized fix seam
def __init__(self, lat, lon, alt_m=0.0, course=0.0, speed=0.0,
             ts=0.0, age_sec=0.0, src="direct_lora"):
    ...
    self.alt_m = float(alt_m)

# 2. Forward it in compute_target
target = GeoPoint(lat=fix.lat, lon=fix.lon, alt_m=getattr(fix, "alt_m", 0.0),
                  speed_mps=fix.speed, course_deg=fix.course)

Root-cause note: the tracker 32-byte packet has no altitude field. The clean fix requires adding alt_m to TrackerPacket (or repurposing reserved bytes) and emitting it from both tracker/base firmware.

High Estimator shadow tilt ignores target altitude and mis-uses base position

orin/wavecam/wavecam/estimator.py:448–454

What is wrong: In predict_output the shadow tilt is computed with both points at (pose.lat, pose.lon) and the target altitude forced to 0.0:

elev = elevation_deg(
    GeoPoint(lat=self._pose.lat, lon=self._pose.lon, alt_m=self._pose.alt_m),
    GeoPoint(lat=self._pose.lat, lon=self._pose.lon, alt_m=0.0),
    dist_m,
)

The horizontal position used is the base, not the estimated target location in ENU space. Elevation therefore depends only on -base.alt_m, not on the actual 3-D geometry to the subject.

Why it matters for aim accuracy: This is shadow logging, not live control, but it is the primary evidence used to validate the estimator before it is allowed to command the camera. Logged tilt_enc_would will be systematically wrong whenever base altitude is non-zero, which can lead to incorrect tuning decisions or false confidence in shadow-mode tracking.

Proposed fix

# Convert current ENU estimate back to lat/lon/alt, or at least use it for Δalt.
# Minimal correct form once target altitude is available:
target_e, target_n = self._x[0], self._x[1]
# ENU horizontal distance already is dist_m; vertical delta must come from fix alt.
# If only 2-D fixes are available, document that tilt shadow is unavailable.
elev = elevation_deg(
    GeoPoint(lat=self._pose.lat, lon=self._pose.lon, alt_m=self._pose.alt_m),
    GeoPoint(lat=target_lat, lon=target_lon, alt_m=target_alt_m),
    dist_m,
)
High iOS heading calibration uses live-base bearing instead of latched-pose geometry

ios/WaveCam/Sources/CalibrateView.swift:362 and orin/wavecam/wavecam/control_calibration.py:549–630

What is wrong: The iOS heading-capture screen sends headingDeg: client.status?.gps?.bearingDeg ?? 0.0. The status-bearing is computed in gps_fix_snapshot using the live base position from get_camera_position(). The backend then stores that value directly as pan_anchor_bearing using the latched pose location. If the two base positions differ, the anchor bearing is inconsistent with the base position used for all future target-bearing calculations.

Why it matters for aim accuracy: All subsequent GPS pan commands compute target_bearing from the latched base, then subtract pan_anchor_bearing. An inconsistent anchor introduces a constant pan bias equal to the bearing error between live and latched base positions. On a moving/wobbling tripod or a base GPS with multipath this can be several degrees.

Proposed fix

// Option A: iOS passes target_lat/target_lon; backend recomputes from latched pose.
// Option B: backend recomputes bearing from latched pose when only bearing_deg is supplied.

# In control_calibration.py heading_lock, before using a supplied bearing_deg:
if bearing_deg is not None and location is not None:
    # Prefer geometry from the latched pose over a client bearing that may
    # have been computed from a different (live) base position.
    lat, lon = location["lat"], location["lon"]
    if target_lat is not None and target_lon is not None:
        bearing_deg = bearing_deg(lat, lon, target_lat, target_lon)
    # Otherwise reject bearing-only captures unless they come with a
    # freshness/uncertainty bound that proves they match the latched pose.

Needs verification: confirm whether gps.bearingDeg in the iOS status model is actually sourced from gps_fix_snapshot as read; the client decoding path and the backend snapshot path both point to the live base position.

Medium No runtime pan/tilt mechanical soft limits are enforced

orin/wavecam/wavecam/camera_pose.py:32–33 and orin/wavecam/wavecam/ptz_visca.py:76–97

What is wrong: PRISUAL_TILT_ENC_MIN/MAX are defined and documented but never used to clamp commands. Pan limits are not represented at all. The only clamp in pan_tilt_absolute is the signed 16-bit wire-format clamp [-0x8000, 0x7FFF].

Why it matters for aim accuracy: A GPS target outside the mechanical envelope (±170° pan, -30°/+90° tilt) produces an encoder command the camera cannot reach. The camera hits the hard stop and stops short; the software believes it has commanded the target and will not correct until the target moves back into range.

Proposed fix

# In ptz_visca.py pan_tilt_absolute, or in camera_pose before conversion:
from wavecam.camera_pose import (
    PRISUAL_TILT_ENC_MIN, PRISUAL_TILT_ENC_MAX,
    PRISUAL_PAN_ENC_PER_DEG,
)
PAN_ENC_MIN = int(round(-170.0 * PRISUAL_PAN_ENC_PER_DEG))  # -2448
PAN_ENC_MAX = int(round( 170.0 * PRISUAL_PAN_ENC_PER_DEG))  # +2448

pan = max(PAN_ENC_MIN, min(PAN_ENC_MAX, int(pan_pos)))
tilt = max(PRISUAL_TILT_ENC_MIN, min(PRISUAL_TILT_ENC_MAX, int(tilt_pos)))
Medium ptz.invert_pan / invert_tilt are ignored for GPS absolute commands

orin/wavecam/wavecam/controller.py:92–95 and orin/wavecam/wavecam/pipeline.py:447–449

What is wrong: invert_pan/invert_tilt only flip velocity-servo and manual joystick directions. GPS absolute pan/tilt encoder targets are computed by CameraPose.bearing_to_pan_encoder and elevation_to_tilt_encoder with no inversion step before sending to pan_tilt_absolute.

Why it matters for aim accuracy: If the camera mount reverses the encoder sense relative to true bearing, the operator will set invert_pan=true to make vision/manual control move correctly. GPS absolute commands will then slew in the opposite direction from the computed target. The two-point calibration rejects negative scale, so the operator cannot calibrate the reversed sense away through scale alone.

Proposed fix

# Apply the same inversion to absolute targets that velocity commands use:
# In pipeline.py _send_absolute_cmd or in gps_pointing compute_target:
if getattr(self.cfg.ptz, "invert_pan", False):
    cmd.pan_enc = 2 * pose.pan_anchor_enc - cmd.pan_enc  # mirror about anchor
if getattr(self.cfg.ptz, "invert_tilt", False):
    cmd.tilt_enc = 2 * pose.tilt_anchor_enc - cmd.tilt_enc

# Cleaner: encode the inversion sign in pan_enc_per_deg during calibration
# and reject a mismatched mount before it reaches the live pointing path.
Medium Base altitude falls back to 0 when the GNSS fix is 2-D only

firmware/direct-lora/src/base/main.cpp:95–96 and orin/wavecam/wavecam/camera_pose.py:139–153

What is wrong: In update_base_gps the base firmware uses base_gps.altitude.isValid() ? ... : 0.0. A 2-D fix (valid lat/lon, no altitude) is therefore latched as altitude 0. The Orin accepts this value via lock_base_position without an altitude-validity gate.

Why it matters for aim accuracy: Same effect as the sea-level target issue: if the camera is actually elevated and the base GPS momentarily reports a 2-D fix, the latched base altitude becomes 0 and every subsequent elevation calculation is biased.

Proposed fix

// In base firmware, add an alt_valid flag to the JSON:
int alt_valid = base_gps.altitude.isValid() ? 1 : 0;
...
"alt_valid":%d,"alt_m":%d,

# In control_calibration.lock_location / capture_calibration base_lock:
if not data.get("alt_valid"):
    return refusal("gps_2d_fix", "Base lock requires a 3-D fix for accurate tilt.")
Medium Status/iOS bearing is computed from live base, not latched pose

orin/wavecam/wavecam/control_snapshots.py:330–345

What is wrong: gps_fix_snapshot calls gps.get_camera_position() to obtain the base position for distance/bearing. It does not use pipeline.pose.lat/lon even when a base location has been locked. This status value drives the iOS operator display and, as shown above, the iOS heading-calibration input.

Why it matters for aim accuracy: The operator sees a bearing/distance that may differ from the geometry the backend uses for actual pointing. If the operator trusts the displayed bearing for manual cueing or calibration, the result is a consistent pan offset.

Proposed fix

# Prefer latched pose; fall back to live base only before a lock exists.
if pipeline.pose.has_base:
    base_lat, base_lon = pipeline.pose.lat, pipeline.pose.lon
    base_age = None  # latched position has no single age; report calibration age
else:
    cam_pos = cam()
    if cam_pos is None:
        return snapshot
    base_lat, base_lon = cam_pos[0], cam_pos[1]
    base_age = cam_age()

dist = haversine_m(base_lat, base_lon, fix.lat, fix.lon)
bearing = bearing_deg(base_lat, base_lon, fix.lat, fix.lon)
Low elevation_deg returns 0° for a directly-overhead target

orin/wavecam/wavecam/gps_geo.py:51–57

What is wrong: The near-zero distance guard returns 0.0 regardless of altitude difference. When base and target share the same horizontal position, the true elevation is ±90°.

Why it matters for aim accuracy: A surfer directly below a cliff-mounted camera would be commanded to the horizon instead of straight down. This is an edge case for the current surf-filming geometry but is mathematically incorrect.

Proposed fix

def elevation_deg(base: GeoPoint, target: GeoPoint, distance_m: float) -> float:
    d_alt = target.alt_m - base.alt_m
    if distance_m <= 1e-6:
        return 90.0 if d_alt > 0.0 else (-90.0 if d_alt < 0.0 else 0.0)
    return math.degrees(math.atan2(d_alt, distance_m))
Low Phone drift monitor ingests magnetic heading, not true heading

ios/WaveCam/Sources/PhoneSensorPublisher.swift:276–283 and orin/wavecam/wavecam/sensor_hub.py:113–215

What is wrong: The iOS publisher sets heading_deg = magneticHeading and only puts trueHeading in the separate true_heading_deg field. The backend drift monitor uses heading_deg for baseline deviation. Magnetic declination is constant for a given location, so deviation-from-baseline cancels it out, but any absolute comparison is offset by local declination (e.g., ~10° in Hawaii).

Why it matters for aim accuracy: Low direct impact because the current calibration flow uses GPS/landmark bearing, not phone heading. If a future client uses method="phone" for heading lock, the captured pan_anchor_bearing will be off by declination and every GPS pan command will be biased by that amount.

Proposed fix

// iOS: send trueHeading in heading_deg; keep magnetic as magnetic_heading_deg if needed.
self.latestHeadingDeg = newHeading.trueHeading >= 0 ? newHeading.trueHeading : nil

# Backend: for phone-method calibration, prefer true_heading_deg if present:
phone_heading = sample.true_heading_deg if sample.true_heading_deg is not None else sample.heading_deg

No Issues Found

  • Bearing wraparound: normalize_180 correctly maps across the 0°/360° seam and the ±180° boundary (orin/wavecam/wavecam/gps_geo.py:45–48).
  • atan2 argument order: bearing_deg uses atan2(sin(Δλ)·cosφ₂, cosφ₁·sinφ₂ − sinφ₁·cosφ₂·cosΔλ), the standard initial-bearing formula, and wraps to [0, 360) (gps_geo.py:36–42).
  • Distance model: haversine_m is great-circle and clamps the asin argument; appropriate for 50–300 m surf-filming ranges (gps_geo.py:27–33).
  • Lead prediction: predict_lead projects along course at speed on the sphere, with a 0.1 m/s floor to avoid amplifying GPS jitter (gps_geo.py:60–73).
  • Relative bearing: CameraPose.bearing_to_pan_encoder computes wrap-safe relative bearing normalize_180(target − anchor) before scaling (camera_pose.py:118–121).
  • Calibration scale sign: two-point pan calibration rejects non-positive scale, preventing reversed GPS slews (camera_pose.py:90–93).
  • Base drift gating: the monitor preserves base_locked=True in unknown/suspect states and only denies GPS authority on confirmed unlocked drift, matching the intended fail-safe behavior (base_drift.py:101–148).
  • Fix freshness gating: live pointing is gated on fix.age_sec < drive_stale_sec and the current calibration session being valid && confirmed; stale fixes do not drive PTZ (pipeline.py:700–728).

Operational note: Severity ratings assume the current surf-filming use case (base near sea level, short-to-medium range, target in the water).
