WaveCam is a vision-guided PTZ camera that keeps you in frame while you foil-surf 50–300 m offshore. A Jetson Orin watches for your orange rashguard plus a person detection, locks on, and drives the camera — you just ride. This guide walks every screen and every tuning control in the app.

How it works

One subject, one camera, one safe path

WaveCam finds you two ways at once and fuses them: a neural net (YOLO11n) spots a person, and a color detector spots your orange rashguard. When both agree, it locks and the camera follows. Everything routes through the Orin so there is exactly one authority for movement and safety.

Cue

Color + person fusion

Your orange rashguard is the standout cue on a blue ocean. YOLO confirms it’s a person, not a buoy. Agreement within a pixel distance = a confirmed target. Score = 0.5 + 0.5×YOLO_conf.

Lock

Lock with hysteresis

Confidence ≥ 0.60 acquires. It only releases below 0.35, so spray and quick turns don’t drop you. Acquire-strict / hold-lenient — by design.

Follow

Smooth servo + zoom

A deadband-and-feed-forward controller pans/tilts without twitching. Optional Cinematic Zoom holds your on-screen size. A shadow Kalman estimator runs in observe-only mode, building its world model for a future pointing upgrade.

Design intent

It watches. It never drives.

Every design choice in WaveCam starts from the same axiom: the camera moves only when you have explicitly armed it and only while a reachable KILL can stop it.

Supervise-only, always

AUTO engages vision-owner or GPS-owner control. But the verify-and-resend loop — which nudges the camera when a GPS slew lands short — is kill-gated and owner-gated: the moment KILL latches, or the moment PTZ ownership changes, any pending resend is cancelled and the target is dropped. A kill landing mid-tick can never be followed by a verifier-issued move.

The Agent panel is advisory-only — it reads telemetry and surfaces logs, but it never moves the camera, runs a shell, or restarts a service.

Architecture direction

Today: vision-owner / GPS-owner arbitration. Vision wins when it has K=5 consecutive locked frames (confidence ≥ 0.60). GPS coarse-points when vision is not locked and GPS is fresh + base-locked. GPS is released (not coasted) on data loss — the camera holds position; it does not continue driving on a stale bearing.

Coming: A shadow Kalman estimator (estimator.shadow=true, enabled=false) currently runs in observe-only mode. It fuses GPS position and vision bearings into a world-frame [east, north, velocity] state at every pipeline tick and logs what it would command — without touching the camera. When the shadow track is validated, flipping estimator.enabled=true will route its bearing output into pointing instead of the current GPS-owner path.

The orange-house incident — why acquire-strict / hold-lenient matters

An early configuration had unlock_threshold at or above lock_threshold. With hysteresis inverted, the unlock branch was unreachable: any color blob that reached the lock level stayed locked permanently. An orange structure near the beach acquired a full lock and the camera parked on it. The hot-config path now refuses any patch where unlock ≥ lock. The YAML loader silently resets to defaults if the file contains an inverted pair rather than bricking the service at the beach.

AUTO  Hand it the wheel

Tap AUTO to request autonomous tracking. The backend decides each frame whether vision or GPS owns the camera. Touching the joystick hands control back to you; tap AUTO again to resume. AUTO refuses while killed — Resume first.

KILL  Always reachable

KILL is a hard latch: it stops all motion immediately and blocks any further movement — including pending verify-and-resend moves — until you tap Resume. The live feed keeps streaming so you can still see. This button is on screen at all times by design.

Calibrate tab · field procedure

Teach it where it’s pointing

Calibration solves the camera-to-world geometry so GPS coarse-pointing lands on target. The solve is stored on the Orin and survives restarts. Run it once per setup spot; re-run Heading + Tilt if you move the tripod. The app walks you through 6 steps in order.

Before you leave — hardware checklist

  • Charge both Wios. Remote goes on you; base plugs into the Orin’s USB-A port.
  • Tripod: camera, Orin, and phone all mounted and stable before you run calibration.
  • Open sky: both Wios need an unobstructed view of the sky. A cold-start first fix can take 2–5 minutes — plan for it.
  • Orin cold-boot with base Wio plugged in — verified safe under systemd-boot (the old U-Boot stall is gone since the NVMe migration). The Wio reader auto-reconnects. No unplug ritual needed.
When the GPS link misbehaves — direct-LoRa check (expand)

Both Wios run the same fixed-config direct-LoRa firmware (firmware/direct-lora/). There is no Meshtastic mesh, no preset to mismatch, and nothing to configure — so a bad link is hardware, power, or firmware, not config. Check this when the remote isn’t heard or the base fix never appears.

Base Wio: acquire the fix on battery (off the Orin’s USB rail — host RF noise can drown the L76K to 0 sats), then connect USB data.

Watch the live GPS state (from a laptop):

ssh orin "curl -s localhost:8088/api/v1/status" # see the gps section # or read the base serial directly: python3 firmware/direct-lora/tools/read_base.py

What good looks like: status.gps shows a fresh target fix (target_sats > 0, target_age_sec low) and the base position latched. If the remote is never heard, check the tracker battery/antenna and that both Wios run the matching direct-LoRa firmware.

The app shows STEP N OF 6 in the header as you progress. Each step has a Confirm or Capture button — tap it only when the expected result is met.

Note: the old separate IMU Level step was removed (2026-06-17) — the rig-phone mounts on its side off the camera and can’t sense pan-axis tilt, so the backend no longer requires a level check. You level the tripod by hand; the wizard goes straight from location to heading. The Tilt — horizon reference step (Step 4) below is unrelated and still captures the camera’s tilt-zero.

Step 2 of 6

Base Lock (GPS)

What it does: Latches the base Wio’s current GPS position as the camera’s fixed ground reference for this session. One capture per session — no need to redo it if the base Wio temporarily loses its fix later.

  1. Watch the Base fix line on the GPS chip — wait until age is stable (a few seconds old, not “—”).
  2. Tap Capture base lock. The Orin latches the position.

If Capture is disabled: base Wio has no fix yet. Move to open sky and wait.

Step 3 of 6

Heading — aim at remote

What it anchors: The pan-motor encoder position paired with the GPS base→remote bearing gives WaveCam its pan-home = “forward” mapping. No magnetometer is used; heading comes entirely from this geometric solve. Every degree of error here shifts the camera’s aim at 100 m by ~1.7 m. Take your time.

This is the accuracy-critical step

Re-run this step any time you move the tripod or the setup faces a different direction.

  1. Put the remote tracker on your person and walk at least 20 m from the tripod, ideally toward the water. Farther is more accurate — 50 m is better than 20 m.
  2. The app shows a live bearing from the base fix to the remote fix. Wait until the bearing value is stable.
  3. Use the joystick to pan/tilt the camera until you are centered in the live feed.
  4. Tap Capture heading. The app gates this on GPS bearing being present — if the button is disabled, wait for both base and remote fixes to be fresh (GPS chip shows distance + bearing).
Step 4 of 6

Tilt — horizon reference

What it anchors: Sets the tilt-zero reference (the encoder position that maps to a level horizon, 0° elevation). The tracker uses this to convert GPS-computed elevation angles into tilt commands. Re-run if the camera is clearly tilted high or low, or when you move the tripod.

  1. Use the joystick to tilt the camera until the horizon (water surface) is level and centered vertically in the live feed.
  2. Tap Capture tilt. The reference elevation is 0.0° (level horizon).
Step 5 of 6

Zoom / FOV curve

What it captures: The current zoom encoder position paired with the known wide-angle FOV (31.5°). This anchors the FOV interpolation curve that the estimator and the GPS-cue boost radius calculation use. The estimator requires a non-empty FOV curve as a gate before shadow mode activates.

  1. Leave the camera at its widest zoom.
  2. Tap Capture zoom. The app records zoom encoder + 31.5° wide reference.

FOV sweep tool — for the full curve

The Calibrate tab captures a single wide-anchor point. For a full zoom-vs-FOV curve (needed for accurate estimator bearing uncertainty at all zoom levels), run tools/calibrate_fov.py on the Orin. Place a color-matched object 15–20 m out. The tool pans from edge to edge, measures the encoder span, and prints a JSON line to POST to /api/v1/calibration/fov. Run once at wide and once at your typical tele position.

Step 6 of 6

Dry-run — verify before water

Do not skip this. Catch a bad heading solve on dry land, not mid-session offshore.

  1. Put the remote tracker on and walk along the beach. You should see: the camera GPS-points and pans to follow you coarsely.
  2. Step into clear view with open sky above. You should see: the lock chip switch to LOCKED as vision acquires your orange rashguard.
  3. Test the kill: tap KILL. You should see: all motion stops instantly. Tap Resume, then AUTO to confirm recovery.
  4. If everything checks out, tap Mark ready and proceed to Live.

Camera GPS-points way off bearing: re-run Step 3 — stand farther away and aim more carefully.

Camera won’t lock up close: confirm Color preset = orange_red in Tune, enable Show mask to verify the rashguard is in the mask.

When to recalibrate

Moved the tripod or changed orientation? Re-run Steps 3 (Heading) and 4 (Tilt) at minimum. Step 2 (Base Lock) must always be run first in any new session.

How GPS pointing works once calibrated

GPS becomes a spotter: it aims the camera at your latest fix so YOLO + color can acquire, then hands off — vision refines the instant it locks. GPS pointing activates only when: fix is fresh (age < gps.stale_threshold_sec — code default 10 s, the rig runs 45 s to cover the remote's 10 s stationary broadcast cadence), calibration is complete, and base is locked. KILL overrides everything.

3 Lock acquire-strict · hold-lenient

Fused confidence needed to acquire a lock. Range: 0.05–0.95. Default: 0.60. A confirmed color+person target scores 0.5 + 0.5×YOLO_conf. Must stay above unlock threshold.
RaiseFewer false locks — waits for a sure, close target.
LowerGrabs eagerly — good in marginal visibility at distance.
ExampleGPS active, lock still not engaging at 200 m → lower to 0.50 or raise gps_boost.
Unlock thresholdHotfusion.unlock_threshold
Confidence below which the lock releases. Range: 0.05–0.95. Default: 0.35. Must stay strictly below lock threshold. Color-only sustain = 0.45, so 0.35 means color alone can hold a lock through spray — but 0.45 cannot start a new lock.

What “confidence” means — the ladder

Lock/unlock compare a fused score, not raw YOLO output. The ladder from code:

  • Color + person matched (blob center within match_dist of person center): 0.5 + 0.5 × YOLO_conf → range ≈ 0.5–1.0. Always acquires above lock=0.60 when YOLO_conf ≥ 0.2.
  • Color-only sustain (CONF_SUSTAIN = 0.45): a color blob (or a person near the last tracked position) while locked. Holds. Does not acquire.
  • Person-only, no color (CONF_PERSON_ONLY = 0.2): a YOLO person with no matching color blob. Never holds, never acquires. Background people are completely ignored.
  • GPS cue boost: when GPS owns the camera and a blob is within the cue radius, it gets +gps_boost (default 0.20), capped at 0.95. A color-only blob at sustain (0.45) + 0.20 boost = 0.65 — enough to cross lock=0.60. This is how GPS shepherds vision onto a far, small target.

v3 Base Drift Monitor v3 Phase 1 · new

When enabled, YOLO runs in track mode (ByteTrack or BoT-SORT) and the fusion layer prefers the same track ID across frames. Helps prevent target swaps when multiple people or orange objects are in frame. Default-off — no behavior change until a tracker is configured + the service restarted.

v3 Estimator (shadow mode) observe-only · do not enable in field yet

The shadow Kalman estimator fuses GPS position and vision bearing into a world-frame [east, north, ve, vn] state. In shadow mode (estimator.shadow=true, estimator.enabled=false) it logs what it would command but never moves the camera. Enabling requires a populated FOV curve (Calibrate Step 5 + the FOV sweep tool) — the estimator will refuse to start without it.

Wio tracker configuration

Direct-LoRa firmware on both Wios

The live GPS path uses the custom firmware in firmware/direct-lora/. There is no Meshtastic mesh, no preset to mismatch, and no app configuration step. Flash once, then power-cycle to re-apply defaults.

Roles

Tracker Wio goes on the subject (you). It reads its L76K GNSS and transmits 32-byte LoRa packets.
Base Wio plugs into the Orin via USB. It receives the tracker packets and emits JSONL over USB serial for DirectRadioGps.

Power matters

The base Wio now has a battery installed. Let it acquire its fix on battery power with clear sky, then connect USB data to the Orin. Keep it off the Orin’s USB rail during acquisition — host USB noise can drive the L76K to 0 sats.

No mesh, no preset, no reconfigure

The custom firmware is compiled with matching radio constants (US915, SF7, BW250). If you change one Wio’s firmware, flash the same radio config to the other. Mismatched firmware/radio settings show up as packet loss or no seq lines at all.

Safety model

Stop is not Kill. Restart is not Resume.

Movement controls are deliberately separate from safety controls. Read the state before repeating a command.

Emergency

KILL

Global latch. Stops all motion now — including any pending verify-and-resend — and blocks future motion until Resume. Feed stays live. Always on screen.

Hold

Stop PTZ

Manual hold — halts pan/tilt/zoom but does not latch. Backend stays reachable; you can move again or re-arm AUTO immediately.

AUTO

Start Auto

Requests autonomous ownership. The arbiter then decides each frame whether vision or GPS drives. Refuses while killed — Resume first. Joystick touch hands back manual control.

Restart

Service

Stops PTZ, then restarts the WaveCam service — used after GPS Ingest goes DOWN or after a Restart-required config change.

The one rule

The automation supervises; it never moves the camera on its own initiative. You — and the KILL button — are always the final authority. Verify-and-resend, GPS absolute slews, and the shadow estimator are all kill-gated: a KILL latch mid-tick immediately clears any pending camera command.
