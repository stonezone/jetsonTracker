Executive summary. The architecture is sound and most documented invariants are honored, but the audit found one critical issue — a runaway-PTZ path on video dropout — plus a cluster of high-severity tracking defects that compound at exactly the 50–300 m mission range: fusion's dropout-grace window is dead code (instant unlock on one bad frame), GPS pointing ignores fix age (aims where the surfer was), the pointing verifier is starved by 20 Hz command resends, and servo gains ignore zoom (limit-cycle hunting at 20×). On the safety side, the in-memory KILL latch can be erased by an automated watchdog restart, and the iOS app ignores ok:false refusals on every non-PTZ command including ARM and the KILL latch display. Fixing C1, H5, H6, and H8 first yields the biggest safety + tracking-quality gain. Pure security findings are recorded in a deprioritized informational section per the operator’s call (single-user personal rig).

Critical

Can physically damage a session. Land this first.

C1Video dropout leaves the camera slewing at its last commanded velocity (runaway PTZ)

TrackingSafety
orin/wavecam/wavecam/pipeline.py:613-617 · capture.py:60-70

Problem — When RTSP drops, the loop only sets state=NO_VIDEO, sleeps 0.1 s, and continues. VISCA velocity commands persist until an explicit stop, so if vision_follow was mid-pan (or cinematic zoom mid-tele) at dropout, the camera runs at that velocity into the hard stop until video returns or a human hits KILL. The ZOMBIE-1 stale-capture guard requires a non-None frame and does not cover this path; no test covers NO_VIDEO stop behavior. Verified directly against the code.

Fix — In the frame is None branch: if cfg.ptz.enabled and the current owner is autonomous (vision_follow/gps_tracker/testbed), send self.ptz.stop() + self._send_zoom("stop") and reset _last_cmd_key before sleeping; optionally release the vision owner after CAPTURE_STALE_SEC. Add a regression test.

High

Safety-invariant gaps and the defects that most directly degrade tracking at 50–300 m.

H1KILL cannot stop an in-flight ARMED agent turn; "KILL is human-only" is prompt-enforced only

SafetyAgent
orin/wavecam/wavecam/control_system.py:137-151,187-191 · agent_session.py:113-126,173-200 · control_api.py:1026-1034

Problem — request_agent_chat captures armed before the turn; agent_kill() only flips ArmState. Nothing tracks or terminates the running claude subprocess, so after KILL an armed turn keeps a bypass-permissions Bash for up to 120 s — and on localhost that shell could curl POST /safety/resume or /system/restart (which clears the in-memory latch, H2). The system prompt's HARD RULES are the only enforcement layer.

Fix — Keep the Popen handle (tracked process group) and have agent_kill() SIGKILL it; give the armed agent a scoped agent token and deny SAFETY/CONFIG/SERVICE actions to that identity at the auth layer; add /system/restart to the forbidden list as defense-in-depth.

H2Automated watchdog restart erases the operator's KILL latch and resumes autonomous motion

SafetyOps
ops/watchdog.sh:81-95,137-171 · orin/deploy/wavecam.service:3-5,21-23 · pipeline.py:127,598-599

Problem — The KILL latch is in-memory only and the service starts unpaused, immediately claiming testbed. The watchdog restarts wavecam.service after two bad gps_reader/health polls — so an operator-KILLed rig with a wedged Wio restarts within two ticks and starts moving again with no human action. Bonus bug: on a rig with gps.enabled=false the gps_reader component is absent, json_path returns __missing__, and the watchdog restarts a healthy service every rate-limit window forever.

Fix — Persist the KILL latch to disk (e.g. next to camera_pose.json) and honor it at boot; make the watchdog check safety.killed before restarting; treat a missing gps_reader component as OK.

H4Neither post-deploy verification nor the watchdog detects a dead vision loop ("zombie rig")

OpsReliability
orin/wavecam/deploy.sh:33-37 · ops/watchdog.sh:20-21,155-180

Problem — deploy.sh verifies only systemctl is-active + a version-SHA match; the watchdog acts only on gps_reader.ok and free disk. CLAUDE.md's own gotcha ("post-deploy verification must check fps>0 while LOCKED — two zombie rigs passed idle checks") is implemented nowhere, so the documented #1 incident class (API answering, vision loop dead) passes the deploy check and survives the watchdog indefinitely. The pipeline already exposes the evidence (components.capture.detail.fps, components.loop).

Fix — deploy.sh: after the SHA check, fail unless components.loop.ok is true and capture.detail.fps > 0. watchdog.sh: add a loop_dead_twice rule (loop not ok, or fps==0 while active, two consecutive runs → rate-limited restart) mirroring the gps_reader rule — but gated on safety.killed being false (H2).

H5Fusion's dropout grace (COASTING) is dead code — one bad frame instantly unlocks

Tracking
orin/wavecam/wavecam/fusion.py:214-237

Problem — When no candidate exists, conf==0 < unlock_threshold sets _locked=False before the coasting check, so elif coasting and self._locked is unreachable (no test reaches COASTING — verified directly). One frame where both the orange blob and the person box are missed (splash/whitewater) instantly unlocks; lock flapping resets the arbiter's 5-frame re-earn, halts cinematic zoom and estimator vision updates, and re-acquiring then needs matched color+person (0.6) — color-only frames (0.45) can never re-hold. This is the dominant lock-churn mechanism at range. The same block also wipes _ema on any single low-conf frame.

Fix — Compute coasting (via _last_seen) before applying the unlock threshold; keep _locked=True while (now - _last_seen) <= lost_grace_sec when no candidate is present; unlock immediately only when a candidate exists with conf below unlock_threshold. Add COASTING-state tests.

H6GPS pointing leads by a fixed 0.65 s but accepts fixes up to 8 s old — aims where the surfer was

TrackingGPS
orin/wavecam/wavecam/pipeline.py:558 · gps_pointing.py:46-56 · gps_direct_lora.py:214

Problem — compute_target(..., lead_s=0.65) is hardcoded while the drive gate admits any fix with age_sec < drive_stale_sec (8.0 s). NormalizedFix.age_sec is available on the very fix being used and is ignored. A foiler at 8 m/s on a 7.9 s-old fix is ~60 m past the aim point — ~22° of pan error at 150 m, several FOV-widths at tele — so GPS coarse-point places vision somewhere it can never re-acquire, defeating the narrow-FOV acquisition strategy. (Both the pipeline and GPS reviews independently flagged this.)

Fix — lead_s = min(fix.age_sec + lead_margin, lead_cap) with lead_margin (~0.65) and lead_cap (~4-5 s, bounding course-extrapolation error) as hot-configurable gps keys; keep the ≥0.1 m/s speed gate in predict_lead.

H7Identical absolute moves re-sent at 20 Hz permanently starve the PointingVerifier

TrackingGPS
orin/wavecam/wavecam/pipeline.py:437-456 · pointing_verifier.py:51-60

Problem — The send gate is if changed or due: with due firing every 0.05 s, so between ~1 Hz GPS fixes the same absolute move is re-sent every 50 ms. Each resend calls record_move(), which resets _issue_t; tick() needs 0.5 s since _issue_t, so verification can never trigger while gps_tracker commands flow — the verify-and-resend mechanism (built for the bench-documented ~390-count overshoot) is dead in exactly the mode it was built for. The 20 Hz stream may also re-trigger the camera's motion profile (post-slew hunting).

Fix — Gate on if changed: (absolute moves need no keepalive; the verifier owns retries) or refresh every 2-3 s ≫ VERIFY_DELAY_SEC; alternatively make record_move() not reset _issue_t when the target is unchanged.

H8Visual-servo gains and deadzone are FOV-independent — limit-cycle hunting at 20× zoom

Tracking
orin/wavecam/wavecam/controller.py:55-60,80-111

Problem — compute() maps normalized image error to VISCA speed with no knowledge of zoom. At full tele (~3.4° HFOV) the 0.05 deadzone is ±0.085°; even pan speed 1 crosses it in ~0.1 s — less than the video+command latency chain — so the camera oscillates around the subject at exactly the long-range zooms this project exists for, while the same mapping is sluggish at wide. The zoom encoder and FOV curve already exist (ptz_state.latest_zoom(), fov_curve) but are never consulted.

Fix — Gain-schedule by FOV: pass current hfov into compute() and scale mapped speed by hfov / hfov_ref, with a degree-denominated deadzone derived from _fov_at_zoom.

H9Base Wio emits an unbounded running-mean position, mathematically defeating the tripod-drift monitor

GPSFirmware
firmware/direct-lora/src/base/main.cpp:81-117 · orin/wavecam/wavecam/base_drift.py · pipeline.py:512-521

Problem — base_lat_mean += (lat - mean)/base_n accumulates over the entire valid-fix run (reset only on fix loss or HDOP>2.5 — hours in open sky). BaseDriftMonitor consumes this mean: after 30 min (n≈1800) a 10 m tripod kick moves the emitted position ~5.5 mm/sample, so the dual trigger (mean>4 m AND trend>2 m over 10 samples) can never fire. The exact failure the monitor exists to catch — silent pointing corruption after a tripod bump — is undetectable.

Fix — Have the firmware emit the instantaneous (or ~30 s-windowed) fix in a separate JSON field alongside the settle mean; feed the drift monitor the raw one and keep the mean for base_lock.

H10iOS ignores ok:false on every non-PTZ POST — ARM display can lie and the KILL latch can wedge

iOSSafety
ios/WaveCam/Sources/WaveCamClient.swift:1095-1163,1211-1220,1524-1566,1727-1808

Problem — Only the PTZ path decodes ok; kill, resume, record start/stop, configHot, setAgentArm, summonAgent, systemRestart, and savePreset treat any 2xx as success, violating the documented "refusals are silent 200s with ok:false" gotcha. Concretely: a refused agent/arm still shows "ARMED — Claude can act"; a refused safety/kill leaves killInFlight/optimisticKilled latched forever, displaying "STOP LATCHED — pan, tilt & zoom halted" while the camera may still be moving; a refused config patch shows as applied.

Fix — Decode WCControlResponse (at least ok/code/message) in all post() callers and route through applyControlResponse; in kill() treat ok==false like the catch branch (clear killInFlight/optimisticKilled, set lastCommandError).

H11Agent chat uses a 5-second HTTP timeout against a 5-30 s synchronous LLM turn

iOSAgent
ios/WaveCam/Sources/WaveCamClient.swift:2075,1753-1762 · orin/wavecam/wavecam/control_api.py:726-731

Problem — post() hardcodes timeoutInterval = 5; /agent/chat blocks through the whole claude -p subprocess with no bytes sent, so URLSession times out on essentially every non-trivial turn. The chat shows "⚠️ Agent chat failed" while the rig-side turn keeps running (and, if armed, keeps acting); the reply is lost and the agentArmed sync in the response never lands.

Fix — Add a per-request timeout parameter to post(); use ~120 s for agent/chat and agent/summon.

H12Joystick and zoom slider flood ptz/velocity/ptz/zoom with unthrottled, unordered POSTs

iOS
ios/WaveCam/Sources/JoystickPad.swift:56-58 · PTZManualController.swift:49-61,144-159 · MergedLiveView.swift:230,328-378

Problem — Every DragGesture tick spawns a detached Task POST; a 2-second stick drag fires ~100+ concurrent takeover:true requests (5 s timeout each) over a tethered/lossy link with no ordering guarantee — a stale high-velocity command can land after a newer low one, producing jerky/overshooting manual PTZ and a delayed release-stop behind the queued backlog, plus needless radio/battery load.

Fix — Throttle in sendVelocity/updateZoom: send immediately if ≥100 ms since the last send, otherwise coalesce to the latest value; always send the final stop (the existing release retry covers it).

H13Watch emergency STOP silently fails on the documented tether-blackhole failure mode

WatchSafety
ios/WaveCam/Sources-Watch/WatchClient.swift:97-100,155-201

Problem — post() fails over to Wi-Fi only on .cannotConnectToHost/.cannotFindHost/.dnsLookupFailed. The code's own comment documents that an absent tether subnet fails as .timedOut — added to GET failover for exactly this reason — but POST still hard-fails on it. Operator hits STOP right as the tether drops → 5 s timeout → false returned, discarded by kill(): no Wi-Fi retry, no UI feedback.

Fix — Allow .timedOut/.networkConnectionLost failover for the idempotent safety endpoints (kill/resume); surface a failed STOP in WatchStatusView (e.g. a "STOP NOT CONFIRMED" banner).

H14CI builds neither the iOS app nor the Wio firmware

CI
.github/workflows/backend-tests.yml:3-11

Problem — The only workflow is path-filtered to orin/wavecam/**. The iOS app (59 tracked files, the operator-safety surface that must keep KILL reachable) and firmware/direct-lora/ (the live GPS source) get zero compile signal; a broken Swift or C++ change surfaces only at manual device-build/flash time.

Medium

Real defects with bounded blast radius: degraded accuracy, misleading state, performance risks, or gaps that bite when other things change.

M1Estimator re-fuses the same cached GPS fix as an independent measurement ~35× per second

Estimator
orin/wavecam/wavecam/pipeline.py:293-295 · estimator.py:260-305

Problem — get_fix() returns the same fix until the next ~1 Hz LoRa packet, and the shadow tick calls update_gps() every frame with no timestamp dedup. Covariance collapses toward zero between fixes, velocity whipsaws on each new fix, and bearing_std_deg in the shadow logs is meaningless — the Plan-3 go/no-go evaluation data is corrupted.

Fix — Store _last_gps_ts and skip update_gps when fix.ts is unchanged.

M2Vision thread takes the control-API lock and builds the full calibration dict every frame

Performance
orin/wavecam/wavecam/pipeline.py:712-715 · control_calibration.py:126-167,479-487

Problem — calibration_state() takes the shared API _lock and allocates a ~20-key dict (copying steps + FOV curve) at 35 fps just to read two booleans. The same lock is held across disk writes and other API handlers, so operator activity can stall the vision loop mid-frame — a direct 30-FPS risk.

Fix — Expose a cheap calibration_gate() returning (valid, confirmed) from an atomic snapshot updated on session state changes, or cache in the pipeline and refresh at ~1 Hz.

M3Estimator vision-bearing update falls back to zoom_enc=0 (widest FOV) on a stale zoom cache

Estimator
orin/wavecam/wavecam/pipeline.py:311-316

Problem — With the camera at tele (3.4°) but FOV assumed wide (~55°), a quarter-frame offset (~0.85° real) is converted to ~14° and fused with r_vis_deg=1.0; one poller hiccup during locked tracking can yank the state tens of metres. The range update already skips on stale zoom; the bearing update doesn't.

Fix — Skip update_vision when _fresh_zoom is None, matching the update_vision_range gate.

M4_scalar_update covariance math is wrong (keeps only the diagonal term of KhP)

Estimator
orin/wavecam/wavecam/estimator.py:429-432

Problem — P[i][j] -= K[i]*h[j]*P[j][j] instead of K[i]·Σₖ h[k]P[k][j]: cross-covariance reduction is dropped and P symmetry breaks, so P can drift non-PSD and bearing_std_deg is unreliable — compounding M1 for the shadow-filter evaluation.

Fix — Use the full (I − Kh)P (the 4×4 matrix shim already exists and is used by update_gps) or Joseph form.

M5Fresh color blobs are associated against YOLO boxes up to 0.6 s old while the camera pans

Tracking
orin/wavecam/wavecam/pipeline.py:665-666 · config.orin.servo.yaml:58

Problem — Cached boxes are reused in image coordinates with no motion compensation; a 10°/s pan at mid-zoom shifts the scene hundreds of px in 0.6 s, so stale boxes either fail to confirm the fresh blob (conf stuck at 0.45, unlockable) or confirm at a phantom offset the EMA then chases — worst exactly during GPS→vision handoff slews.

Fix — Shorten box_ttl_sec to ~0.2 s, or shift cached boxes by the encoder delta since capture (ptz_state + fov_curve give px/deg), or bypass TTL reuse while a non-stop PTZ command is active.

M6Fusion._last_track_id is never cleared on lock loss — recycled tracker IDs can steal the lock

Tracking
orin/wavecam/wavecam/fusion.py:135-144,157,226-237

Problem — After a long dropout, _prefer_track_id overrides continuity and proximity in favor of whichever new person inherited the stale ByteTrack ID — e.g. locking a beach walker instead of the subject.

Fix — Set _last_track_id = None in the if not coasting: reset branch alongside _ema = None.

M7Overlay drawing + JPEG encode run every frame even with zero preview clients

Performance
orin/wavecam/wavecam/pipeline.py:798-823 · overlay.py:27-33

Problem — Per frame: frame.copy(), np.zeros_like tint + boolean index + addWeighted, box/text drawing, and cv2.imencode — several ms of the 28 ms budget, pure observability with no MJPEG viewers (the normal field state), and it silently becomes the bottleneck if camera.source ever points at the 1080p stream.

Fix — Track active preview clients in the web layer and skip annotate/encode (or encode every Nth frame) when none are connected; at minimum reuse a preallocated tint buffer.

M8Firmware speed/course validity flags are dropped — course 0° (due north) is reported as real data

GPSFirmware
firmware/direct-lora/src/base/main.cpp:178-190 · orin/wavecam/wavecam/gps_direct_lora.py:218-219

Problem — The base JSON emits only the fix flag; PKT_FLAG_SPEED_VALID/PKT_FLAG_COURSE_VALID never cross the wire, and the parser decodes speed_cm_s/course_cdeg unconditionally. When fix is valid but course isn't (slow drift), the memset 0 reads as course=0° and predict_lead leads the subject north — small today, a direct pointing error once H6 grows lead to seconds.

Fix — Emit "spd_ok"/"crs_ok" in the base JSON; parser nulls course / zeroes speed when unset.

M9Remote fix quality (hacc_cm, sats) never gates pointing — a 30 m fix drives pan like a 1 m fix

GPS
firmware/direct-lora/src/tracker/main.cpp:126-127 · orin/wavecam/wavecam/gps_stub.py:17

Problem — The base emits hacc_cm but no Python code reads it; NormalizedFix has no accuracy field. A degraded post-wipeout reacquisition fix (~30 m) is ~11° of error at 150 m — off-frame at tele — and the bearing cue still claims a fixed 5° uncertainty.

Fix — Add h_acc_m to NormalizedFix, parse hacc_cm/100, and above a threshold either withhold drive authority or inflate bearing_uncertainty_deg and widen zoom.

M10Calibration API's base_locked reports "has a base position", not the actual drift lock

GPSAPI
orin/wavecam/wavecam/control_calibration.py:147-149 · pipeline.py:709

M11Missing sample altitude is coerced to 0.0 and averaged into the locked base altitude

GPSCalibration
orin/wavecam/wavecam/control_calibration.py:374,442

M12Heading-uncertainty model is selected by payload field presence, not by method — close-range GPS capture can pass the 2° budget
